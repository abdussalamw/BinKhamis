# مرجع نظام حلقات برو (Halagat Pro) - معمارية الفصل التام

هذا المستند يمثل المرجع التقني الشامل لهيكلية النظام بعد تحديثها لدعم "الفصل المطلق" للبيانات بين المجمعات التعليمية.

## 1. الهيكلية التقنية (High-Level Architecture)

يعتمد النظام معمارية **Multi-Tenancy عبر PostgreSQL Schemas**، حيث يتم منح كل مجمع تعليمي **سكيما (Schema) مستقلة** داخل قاعدة بيانات PostgreSQL واحدة. هذا يوفر عزلاً منطقياً كاملاً للبيانات بين المجمعات مع الحفاظ على سهولة الإدارة.

### المكونات الرئيسية:
*   **القاعدة المركزية (Central Database)**: تحتوي على بيانات المنصة العامة، سجل المجمعات المشتركة، وبيانات "رئيس المنصة" (مخزنة في schema `public`).
*   **سكيما المجمعات (Tenant Schemas)**: سكيما ديناميكية يتم إنشاؤها تلقائياً لكل مجمع عند التسجيل (مثال: `school_a1b2c3d4-e5f6...`)، تحتوي على الطلاب، المعلمين، الحلقات، والتقارير.
*   **نظام التوجيه (IdentifyTenant Middleware)**: برمجية وسيطة في "الباك إند" تقوم بتبديل `search_path` لجلسة PostgreSQL إلى السكيما الصحيحة بناءً على رأس الطلب `X-School-ID`.

### ملاحظة حول العزل:
```
PostgreSQL Schema ≠ قاعدة بيانات منفصلة
✓ عزل منطقي: كل مجمع في سكيما منفصلة مع جداول خاصة به
✓ عزل أداء: استعلامات مجمع (أ) لا تؤثر على مجمع (ب)
✗ ليس عزل فيزيائي: جميع المجمعات في نفس خادم قاعدة البيانات (نفس ملفات البيانات)
```

## 2. نظام الفصل والأمان (Security & Isolation)

تم تنفيذ الفصل عبر 3 طبقات أمان لضمان عدم تداخل البيانات:

### طبقة 1 - التوجيه (IdentifyTenant Middleware)
1. **التعرف على المجمع**: يتم إرسال `X-School-ID` في كل طلب API من الواجهة الأمامية.
2. **تبديل السكيما**: يقوم الميدلوير بـ `SET search_path` لاتصال قاعدة البيانات إلى سكيما المجمع المطلوب.
3. **التوافق مع الإصدارات القديمة**: يدعم النظام أسماء السكيما القديمة (أول 8 أحرف من UUID) كـ fallback للتوافق الرجعي.

### طبقة 2 - التحقق من التوكن (ApiTokenMiddleware)
1. **التحقق من هوية المجمع**: بعد العثور على المستخدم، يتم التحقق من أن `school_id` الخاص به يطابق `X-School-ID` المرسل في الطلب. إذا لم يتطابقا، يُرفض الطلب نهائياً (403).
2. **صلاحية التوكن**: التوكن له صلاحية محددة (30 يوماً). إذا انتهت صلاحيته، يُطلب من المستخدم إعادة تسجيل الدخول.
3. **تسجيل الدخول الآلي**: بعد التحقق، يتم تسجيل دخول المستخدم تلقائياً عبر `auth()->login()`.

### طبقة 3 - نطاق العموم (BelongsToSchool Trait)
1. **نطاق عمومي (Global Scope)**: كل استعلامات Eloquent تُصفّى تلقائياً بـ `school_id`.
2. **الحماية من التسريب**: في حال عدم وجود مستخدم مسجل الدخول، يُعاد مجموعة فارغة (وليس جميع البيانات من جميع المجمعات).

### ملخص طبقات العزل:
```
طبقة 1: IdentifyTenant Middleware  →  search_path للسكيما الصحيحة
طبقة 2: ApiTokenMiddleware          →  التحقق من X-School-ID + صلاحية التوكن
طبقة 3: BelongsToSchool Trait       →  Global scope على school_id
```

## 3. دورة حياة المجمع الجديد (Provisioning Flow)

عند إضافة مجمع تعليمي جديد من لوحة تحكم "رئيس المنصة":
1. يتم تسجيل المجمع في القاعدة المركزية (جدول `schools` في schema `public`).
2. يتم إنشاء **سكيما جديدة** باسم فريد (مثال: `school_a1b2c3d4-e5f6-...` باستخدام UUID الكامل).
3. يتم تشغيل "الترحيلات" (Migrations) آلياً لبناء الجداول داخل السكيما الجديدة.
4. يتم إنشاء حساب "مدير المجمع" داخل سكيما المجمع الجديد مباشرة.
5. يتم تعيين صلاحية للتوكن (30 يوماً) قابلة للتجديد عند كل تسجيل دخول.

## 4. المسميات الوظيفية المعتمدة (Standardized Roles)

تم توحيد المسميات المهنية في كامل النظام لضمان الاحترافية:
*   **رئيس المنصة (Platform Owner)**: المتحكم الكامل في المنصة والمجمعات. مخزّن في القاعدة المركزية (central).
*   **مدير المجمع (School Manager)**: المسؤول الأول عن المجمع التعليمي (role: `admin`).
*   **مدير الشؤون الإدارية (Admin)**: مسؤول العمليات اليومية والطلاب (role: `admin`).
*   **المشرف التعليمي (Supervisor)**: يتابع الحلقات والمعلمين (role: `supervisor`).
*   **معلم حلقة (Teacher)**: يدير الحلقات والتحضير (role: `teacher`).
*   **طالب (Student)**: يتتبّع حضوره وتقاريره (role: `student`).

ملاحظة: نظام الصلاحيات يستخدم **حقل `role` النصي** حالياً. مكتبة Spatie مثبتة في الكود ولكنها غير مفعلة في ميدلوير التحقق من الأدوار.

## 5. دليل المطور (Developer Notes)

### إضافة حقل جديد في قاعدة البيانات:
*   إذا كان الحقل يخص المجمعات (مثل إضافة حقل في جدول الطلاب)، يجب التأكد من تشغيل الترحيل على كافة سكيما المستأجرين.
*   يمكن استخدام الأمر التالي لتشغيل الترحيلات على مجمع محدد:
    `php artisan migrate --database=tenant --path=database/migrations/tenant`
*   لإضافة حقل في القاعدة المركزية:
    `php artisan migrate --database=central`

### التواصل مع API:
يجب التأكد من أن جميع الطلبات تحتوي على الرأس التالي:
`X-School-ID: [UUID-OF-SCHOOL]`

### آلية التوكن والمصادقة:
- التوكن يُنشأ مع صلاحية 30 يوماً (قابلة للتعديل في `AuthController::respondWithToken`).
- عند انتهاء الصلاحية، يعيد النظام `401 - Token expired` ويجب على المستخدم إعادة تسجيل الدخول.
- لتسجيل الخروج: يتم مسح التوكن (`api_token = null`) عبر الـ API `/auth/logout`.
- يتم تخزين التوكن مشفراً (SHA-256) في قاعدة البيانات.

### إنشاء ميجريشن جديد:
```bash
# إضافة حقل لجدول الـ users:
php artisan make:migration add_field_to_users_table

# بعد إنشاء الملف، شغّل الميجريشن على central:
php artisan migrate --database=central
```

## 6. الرؤية المستقبلية

*   **التوسع لقواعد بيانات منفصلة**: يمكن مستقبلاً نقل كل سكيما إلى قاعدة بيانات PostgreSQL منفصلة لتحقيق عزل فيزيائي كامل.
*   **النطاقات الفرعية (Subdomains)**: النظام مهيأ لدعم النطاقات الفرعية؛ حيث يمكن ربط كل مجمع برابط خاص به (مثل `khamis.halagat.pro`).
*   **توحيد الصلاحيات**: استخدام مكتبة Spatie بالكامل أو إزالتها لتبسيط نظام الصلاحيات.
*   **تحسين المصادقة**: دعم Laravel Sanctum الأصلي لتوفير refresh tokens وmulti-device sessions.
